## 1. Project overview (plain English)

AutoDoc.ai is a multi-tenant AI document extraction system.
A user logs in, uploads a document (PDF or image), selects a Template (a list of fields to extract),
and the backend:

1) extracts text (embedded PDF text when available; otherwise Google Vision OCR)
2) cleans the text
3) asks an OpenAI model to produce JSON matching the template fields
4) assigns a confidence score per field
5) stores the results in PostgreSQL as Extraction rows

The frontend is a React SPA with a public marketing site and role-based dashboards.

### Technologies (grouped)

Backend:
- FastAPI + Uvicorn
- SQLAlchemy ORM + Alembic migrations
- PostgreSQL (JSONB + native ENUM for roles)
- Passlib hashing + python-jose JWT
- PyMuPDF + Pillow + Google Cloud Vision (OCR)
- OpenAI SDK (LLM extraction)

Frontend:
- React + React Router + Vite
- Tailwind CSS
- Stripe SDKs included (UI only currently)

Infra:
- Local dev workflow, .env configuration (no Dockerfile / no CI/CD in repo)

## 2. Glossary (the core objects)

If you understand these, the whole system becomes predictable:

- Client: a tenant/company
- User: a person who logs in (belongs to a Client except super_admin)
- Role: user, business_admin, enterprise_admin, super_admin
- Plan: subscription entitlements (max users, can_manage_templates)
- Template: extraction blueprint (global or client-specific)
- Field: one extractable value definition inside a Template
- Document: an uploaded file + processing status
- Extraction: one extracted field value for one Document

## 3. System diagrams

### 3.1 Request/response flow

`
React UI
  → (fetch + Authorization: Bearer JWT)
FastAPI
  → Middleware: CORS
  → Middleware: API logging → api_logs
  → Dependency: get_current_user (JWT → user)
  → Route handler
  → CRUD layer (SQLAlchemy)
  → Services (OCR + LLM pipeline when processing)
PostgreSQL
`

### 3.2 Extraction pipeline flow

`
Upload
  → OCR text (embedded or Google Vision)
  → Clean text
  → OpenAI extraction (JSON)
  → Confidence scoring
  → Save Extraction rows
  → Return results
`

## 6. Backend API (how to think about requests)

Typical request lifecycle:

1) CORS middleware
2) API logging middleware
3) Dependencies: DB session + current user (JWT)
4) Optional role checks (
equire_superadmin, 
equire_template_management)
5) Route handler
6) CRUD/services
7) Response returned
8) Log row written to `api_logs`

Endpoint groups:

- Auth: /auth/*
- Tenants: /clients/*
- Users: /users/*
- Billing: /plans/*, /subscriptions/*, /payments/*
- Templates/Fields: /templates/*, /fields/*
- Documents: /documents/*
- Extractions: /extractions/*

For the full endpoint-by-endpoint breakdown (schemas + step-by-step flow), see the appendix.

## 7. Security (what exists, where)

- Password hashing: `backend/app/core/security.py` (Passlib)
- JWT: `backend/app/core/jwt.py` + enforced via `backend/app/api/deps.py`
- Authorization (roles): `require_superadmin`, `require_template_management` in `backend/app/api/deps.py`
- Tenant isolation: route-level client_id checks across resources
- Template access control: `backend/app/core/permissions.py`
- Upload hardening: UUID filenames in `backend/app/api/routes/documents.py`
- SQL injection prevention: SQLAlchemy bound parameters; Alembic uses sa.text(..., params)
- Missing today: rate limiting, refresh tokens, token revocation list

---

## 8. End-to-end pipeline (one user action)

User uploads and processes a document:

1) POST /documents/upload saves file + creates Document row
2) POST /documents/{id}/process runs OCR + LLM extraction and writes Extractions
3) GET /documents/{id}/extractions or /extractions/summary returns results

Key files involved:

- `backend/app/api/routes/documents.py`
- `backend/app/services/document_pipeline.py`
- `backend/app/services/ocr.py`
- `backend/app/services/text_cleanup.py`
- `backend/app/services/extraction_context.py`
- `backend/app/services/llm_extraction.py`
- `backend/app/crud/crud_extraction.py`

Full line-by-line pipeline trace is in the appendix.

## 12. Trade-offs & improvements

Biggest trade-off: processing is synchronous inside the HTTP request.
Improvements to prioritize:

- Background processing (Celery/Redis)
- Rate limiting
- Token refresh + revocation
- Real refund audit trail (don’t delete payments)
- Build the missing dashboard UIs on top of existing APIs
- Add pytest/Vitest suites
- Add Docker + CI pipeline

## 3. Database Design & Schema

The schema is defined in Python by SQLAlchemy 2.x models (`backend/app/models/*.py`) and
materialized by the Alembic migrations under `backend/alembic/versions/`. The database
is **PostgreSQL** (the migrations use Postgres-specific features: `JSONB` and a native
`ENUM` type for the user role).

### 3.1 Table `clients` (file: `app/models/client.py`)

Purpose: represents a **tenant** / company. All per-tenant data (users, templates,
documents, subscriptions, payments, logs) links back to a client row.

| Column       | Type                       | Constraints                                 | Why this type / why this column |
| ------------ | -------------------------- | ------------------------------------------- | ------------------------------- |
| id           | INTEGER                    | PK, indexed                                 | INT autoincrement is enough for expected scale; cheap join key. |
| name         | VARCHAR(255)               | NOT NULL, indexed                           | Display name. 255 chars is the legacy default for `VARCHAR` and is comfortably large for any company name. Indexed because it’s searched/listed. |
| company_name | VARCHAR(255)               | nullable                                    | Some tenants give only one name; kept separate to allow legal company name later. |
| email        | VARCHAR(255)               | nullable                                    | Billing contact email. Not unique because multiple clients can share an admin email. |
| api_key      | VARCHAR(255)               | UNIQUE, indexed, nullable                   | Random token (`secrets.token_urlsafe(32)`) for server-to-server API calls. UNIQUE so a key identifies exactly one tenant; indexed for fast lookup. Kept nullable so legacy rows without an API key don’t break. |
| created_at   | TIMESTAMPTZ                | default `now()`                             | Audit/analytics. `WITH TIME ZONE` because the app stores UTC and shows local time on the client side. |

Relationships (enforced by SQLAlchemy `relationship(... cascade="all, delete-orphan")`):
- `clients` 1 – N `users` (`users.client_id → clients.id`)
- `clients` 1 – N `templates` (`templates.client_id → clients.id` — nullable for global templates)
- `clients` 1 – N `documents`
- `clients` 1 – N `subscriptions`
- `clients` 1 – N `payments`
- `clients` 1 – N `api_logs`

### 3.2 Table `users` (file: `app/models/user.py`)

Purpose: account rows. Holds login identity, hashed password, and role.

| Column          | Type                           | Constraints                 | Why |
| --------------- | ------------------------------ | --------------------------- | --- |
| id              | INTEGER                        | PK, indexed                 | Surrogate key. |
| email           | VARCHAR(255)                   | NOT NULL, UNIQUE, indexed   | Primary login identifier. UNIQUE to prevent duplicate accounts. Indexed because every login query filters by `lower(email)`. |
| username        | VARCHAR(255)                   | nullable                    | Display name; optional. |
| hashed_password | VARCHAR(255)                   | NOT NULL                    | Passlib output. Never stores plaintext. 255 chars is enough for `pbkdf2_sha256`/`bcrypt` digest strings. |
| role            | ENUM user_role                 | NOT NULL, default `'user'`  | Postgres enum created by migration `e3b9c2a4d1f7` and renamed in `f1a2b3c4d5e6`. Values: `user`, `super_admin`, `business_admin`, `enterprise_admin`. Native enum is used (instead of VARCHAR) for integrity — only the 4 legal values are writable. |
| client_id       | INTEGER                        | FK `clients.id`, indexed, nullable | Tenant link. Nullable because a `super_admin` may have no client. Indexed because every tenant-scoped query filters by it. |
| created_at      | TIMESTAMPTZ                    | default `now()`             | Audit. |

Why `email` is stored case-insensitively (not with a Postgres `CITEXT` column): the
codebase normalizes via `lower(trim(email))` in `_normalize_email` and queries with
`func.lower(User.email)`; doing that in SQLAlchemy is portable (the original Alembic
migration was written for a generic `VARCHAR(50)` → `user_role` enum upgrade path on
a non-CITEXT base).

### 3.3 Table `plans` (file: `app/models/plan.py`)

Purpose: subscription plan catalog. Controls pricing and feature entitlements.

| Column                | Type         | Constraints           | Why |
| --------------------- | ------------ | --------------------- | --- |
| id                    | INTEGER      | PK                    | Key. |
| name                  | VARCHAR(100) | NOT NULL              | "Business", "Enterprise". |
| monthly_price         | INTEGER      | NOT NULL, default 0   | Money is stored as INTEGER (cents). Float/double are forbidden for money because of rounding; using an integer amount avoids that class of bug. |
| max_users             | INTEGER      | NOT NULL, default 1   | Tenant seat cap. Enforced at `create_user` via `ensure_client_can_add_user`. |
| allow_creation        | BOOLEAN      | default TRUE          | Whether the plan allows creating new resources. |
| is_active             | BOOLEAN      | default TRUE          | Soft archive — `/plans` returns only active plans. |
| created_at            | TIMESTAMPTZ  | default `now()`       | Audit. |
| can_manage_templates  | BOOLEAN      | default FALSE, added in migration `d4c49f63235c` | Gate that decides whether an `ENTERPRISE_ADMIN` can add/edit/delete templates for their client. Default FALSE so upgrading to enterprise is an explicit decision. |

Relationships: 1 – N to `subscriptions`.

### 3.4 Table `subscriptions` (file: `app/models/subscription.py`)

| Column          | Type         | Constraints                 | Why |
| --------------- | ------------ | --------------------------- | --- |
| id              | INTEGER      | PK                          | |
| client_id       | INTEGER      | NOT NULL, FK `clients.id`, indexed | A subscription is owned by exactly one tenant. |
| plan_id         | INTEGER      | NOT NULL, FK `plans.id`, indexed   | Which plan is active. |
| status          | VARCHAR(50)  | NOT NULL, default `'active'`| State machine: `active`, `canceled`, `past_due`. Stored as VARCHAR instead of native enum to keep later additions easy without migrations. |
| start_date      | TIMESTAMPTZ  | nullable                    | When the subscription began. |
| end_date        | TIMESTAMPTZ  | nullable                    | Set when canceled. |
| created_at      | TIMESTAMPTZ  | default `now()`             | Audit. |

Uniqueness rule is enforced in the route (`/subscriptions` blocks creating a second
active subscription for the same client), not by the DB. This was a time trade-off
(see section 10).

### 3.5 Table `payments` (file: `app/models/payment.py`)

| Column           | Type        | Constraints                            | Why |
| ---------------- | ----------- | -------------------------------------- | --- |
| id               | INTEGER     | PK                                     | |
| subscription_id  | INTEGER     | NOT NULL, FK `subscriptions.id`, indexed | Payment is tied to a subscription event. |
| client_id        | INTEGER     | NOT NULL, FK `clients.id`, indexed     | Denormalized for quick tenant filtering. |
| status           | VARCHAR(50) | NOT NULL, default `'pending'`          | `pending` / `paid` / `failed`. Mock payments are created automatically with `status='paid'` in `/subscriptions`. |
| created_at       | TIMESTAMPTZ | default `now()`                        | Audit/sort. |

### 3.6 Table `templates` (file: `app/models/template.py`)

Purpose: a blueprint that defines what fields the LLM should extract from a document.
Templates can be **global** (`is_global=TRUE`, `client_id IS NULL`) — everyone sees them
— or **per-client**.

| Column      | Type          | Constraints                         | Why |
| ----------- | ------------- | ----------------------------------- | --- |
| id          | INTEGER       | PK                                  | |
| client_id   | INTEGER       | FK `clients.id`, indexed, nullable  | NULL when the template is global. Indexed because tenant filtering is the hot query. |
| name        | VARCHAR(255)  | NOT NULL                            | Human name (e.g. "CV", "Invoice", "بطاقة هوية لبنانية"). |
| description | TEXT          | nullable                            | Free text; injected into the LLM prompt as additional context. TEXT (no length limit) because descriptions can be multi-paragraph. |
| is_global   | BOOLEAN       | default FALSE                       | Flag used in `list_templates_for_user`, `require_template_management` and `ensure_user_can_use_template`. |
| created_at  | TIMESTAMPTZ   | default `now()`                     | Audit. |

Relationships: N – 1 `client`; 1 – N `fields` (cascade delete); 1 – N `documents`.

### 3.7 Table `fields` (file: `app/models/field.py`)

Purpose: one row per extractable attribute of a template. Drives the prompt in
`app/services/llm_extraction.py::_build_messages`.

| Column      | Type         | Constraints                          | Why |
| ----------- | ------------ | ------------------------------------ | --- |
| id          | INTEGER      | PK                                   | |
| template_id | INTEGER      | NOT NULL, FK `templates.id`, indexed | Parent template. |
| name        | VARCHAR(255) | NOT NULL                             | Canonical JSON key the LLM must return (`invoice_number`, `full_name`, `الاسم`, ...). Kept as VARCHAR rather than an enum because it is data-driven. |
| label       | VARCHAR(255) | nullable (added in migration `c2a1b3c4d5e6`) | Human-facing label shown on the document. Used to help the LLM identify where to look. |
| description | TEXT         | nullable (added in migration `a1b2c3d4e5f6`) | Free-form guidance inserted into the prompt ("Return digits only if the card…"). |
| field_type  | VARCHAR(50)  | nullable                             | Drives confidence scoring heuristics: `date`, `number`/`integer`/`currency`, `email`, `phone`, else `text`. |
| required    | BOOLEAN      | default FALSE                        | If missing at extraction time, confidence becomes 0.0 (vs 0.1 for optional). |
| created_at  | TIMESTAMPTZ  | default `now()`                      | Audit. |

Relationships: N – 1 `template`; 1 – N `extractions` (cascade delete — deleting a field
deletes its historical extraction rows too).

### 3.8 Table `documents` (file: `app/models/document.py`)

| Column       | Type          | Constraints                           | Why |
| ------------ | ------------- | ------------------------------------- | --- |
| id           | INTEGER       | PK                                    | |
| client_id    | INTEGER       | NOT NULL, FK `clients.id`, indexed    | Tenant scoping. |
| template_id  | INTEGER       | FK `templates.id`, indexed, nullable  | Nullable so that documents uploaded without a template can exist historically; processing endpoints refuse to run if it is NULL. |
| file_url     | VARCHAR(2048) | nullable                              | Relative or absolute path to the uploaded file on disk. 2048 chars is URL-safe (RFC guidance). |
| status       | VARCHAR(50)   | NOT NULL, default `'uploaded'`        | State machine: `uploaded` → `processing` → `done`/`failed`. String instead of enum for ease of evolution. |
| created_at   | TIMESTAMPTZ   | default `now()`                       | Audit / ordering (`list_documents_for_user` orders by this). |
| processed_at | TIMESTAMPTZ   | nullable                              | Populated by `set_document_status` when status becomes `done`. |

Relationships: N – 1 `client`, N – 1 `template`, 1 – N `extractions` (cascade).

### 3.9 Table `extractions` (file: `app/models/extraction.py`)

| Column       | Type         | Constraints                            | Why |
| ------------ | ------------ | -------------------------------------- | --- |
| id           | INTEGER      | PK                                     | |
| document_id  | INTEGER      | NOT NULL, FK `documents.id`, indexed   | Owning document. |
| field_id     | INTEGER      | FK `fields.id`, indexed, nullable      | Usually the Field definition; nullable in case the LLM returns something not in the template schema. |
| value_text   | TEXT         | nullable                               | String form of the extracted value. TEXT because it may be a long paragraph (e.g. CV summary, work_experience). |
| value_json   | JSONB        | nullable                               | Structured value (list, dict, number) as returned by the LLM. JSONB so we can query/filter without reparsing. |
| confidence   | FLOAT        | nullable                               | Output of `calculate_confidence(...)`: 0.0–1.0. Nullable because a legacy row might not have it. |
| created_at   | TIMESTAMPTZ  | default `now()`                        | Audit. |

### 3.10 Table `api_logs` (file: `app/models/api_log.py`)

| Column            | Type          | Constraints                       | Why |
| ----------------- | ------------- | --------------------------------- | --- |
| id                | INTEGER       | PK                                | |
| client_id         | INTEGER       | FK `clients.id`, indexed, nullable | Derived from the JWT if any; NULL for unauthenticated traffic. |
| endpoint          | VARCHAR(1024) | NOT NULL                          | Stored as `"METHOD /path"` for easy grouping. |
| status_code       | INTEGER       | nullable                          | HTTP status returned. |
| request_payload   | JSONB         | nullable                          | Truncated JSON body (max 10 000 bytes) if method was POST/PUT/PATCH and the body parsed. |

| created_at        | TIMESTAMPTZ   | default `now()`                   | Insert time. |

### 3.11 Indexes (full list)

Created by migration `9c7abaf1781f_create_initial_tables.py` and kept in sync:

  server-to-server.
- `ix_clients_id`, `ix_clients_name` — listing/search.
- `ix_plans_id`.
- `ix_users_email` UNIQUE — login by email.
- `ix_users_id`, `ix_users_client_id` — tenant scoping.
- `ix_api_logs_id`, `ix_api_logs_client_id` — tenant-scoped log fetches.
- `ix_subscriptions_id`, `ix_subscriptions_client_id`, `ix_subscriptions_plan_id` —

- `ix_templates_id`, `ix_templates_client_id` — tenant-scoped listing.
- `ix_documents_id`, `ix_documents_client_id`, `ix_documents_template_id` — tenant-

### 3.12 Triggers, stored procedures, views

None. The only "server-side logic" is `server_default=func.now()` on `created_at`
columns and the native `user_role` enum. Enum transitions happen in migrations:

- `e3b9c2a4d1f7_user_role_enum.py` creates the type with values including spaces
  (`'super admin'`, etc.), then normalizes all legacy `VARCHAR` role values, then runs
  `ALTER TABLE users ALTER COLUMN role TYPE user_role USING role::user_role`.
- `f1a2b3c4d5e6_user_role_enum_underscores.py` renames the enum values to use
  underscores (`'super_admin'`, `'business_admin'`, `'enterprise_admin'`) so that the
  Python `UserRole` enum values map 1-to-1.

The seed migrations (`b7c8d9e0f1a2`, `e0f1a2b3c4d5`, `f0e1d2c3b4a5`, `g9f8e7d6c5b4`,
`c8d9e0f1a2b3`) insert/delete rows in `templates` and `fields` using parameter-bound
`sa.text(...)` queries. Each has a `_has_column(conn, table, column)` helper that
checks `information_schema.columns` so the migrations are resilient to being run
against older schemas (e.g. before `fields.label` or `fields.description` existed).

### 3.13 Relationships summary

```
clients 1─┬─* users
          ├─* templates  (template.client_id nullable ⇒ global)
          ├─* documents ─* extractions ─1 fields ─1 templates
          ├─* subscriptions ─1 plans
          │                 └─* payments
          └─* api_logs
```

Deletion cascades (ORM `cascade="all, delete-orphan"`):
- Deleting a `client` removes its users, templates, documents, subscriptions, payments,
  api_logs.
- Deleting a `template` removes its fields (and cascade-deletes their extractions).
- Deleting a `subscription` removes its payments.
- Deleting a `document` removes its extractions.
- Deleting a `field` removes its historical extractions.

## 4. Backend API Endpoints

All endpoints are mounted by `backend/app/api/router.py` under the FastAPI root; no
global prefix. Each router declares its own prefix (`/auth`, `/users`, `/clients`,
`/templates`, `/fields`, `/documents`, `/extractions`, `/plans`, `/subscriptions`,
`/payments`). Every protected route receives the current user via
`Depends(get_current_user)` or one of the role-gating deps in `app/api/deps.py`.

Request bodies are validated against the Pydantic schemas in `app/schemas/`. FastAPI
returns a `422 Unprocessable Entity` with the field-level errors when validation fails.

Error response format (FastAPI default): `{"detail": "<message>"}` or
`{"detail": [{...validation errors...}]}`. The frontend's `Login.jsx` handles both.

### 4.1 Auth (`/auth`) — `app/api/routes/auth.py`

- **POST `/auth/register`** — *Admin-only user creation* (Create)
  - Body: `UserCreate` (`email: EmailStr`, `password` min 8 / max 72, `username?`,
    `role = user`, `client_id?`).
  - Auth: `Depends(get_current_user)`.
  - Flow: checks no duplicate via `get_by_email`; if caller is `SUPER_ADMIN` then
    `payload.client_id` is required, otherwise caller's `client_id` is forced — a
    non-superadmin cannot create a user in another tenant. Calls `create_user`, which
    calls `ensure_client_can_add_user` to enforce `Plan.max_users`.
  - Response: `UserOut`.

- **POST `/auth/signup/business`** (Create)
  - Body: `SignupRequest` with `client_type = "business"`.
  - Auth: none.
  - Flow: `_validate_client_type(payload, "business")` → `_signup_with_role(...,
    UserRole.BUSINESS_ADMIN, db)`. Inside `_signup_with_role`:
    1. `get_by_email` → 400 if duplicate.
    2. `create_client(db, name=..., company_name=..., email=..., commit=False)` —
       inserts a tenant row.
    3. `create_user(db, ..., role=BUSINESS_ADMIN, client_id=client.id,
       enforce_limits=False, commit=False)` — creates the admin user.
    4. `db.commit()`, `db.refresh(client)`, `db.refresh(user)`.
    5. On any exception: `db.rollback()` and raise 500.
  - Response (201): `SignupResponse` with `client` + `user`.

- **POST `/auth/signup/enterprise`** — same, with `UserRole.ENTERPRISE_ADMIN`.

- **POST `/auth/login`** (Read) — used as the token URL.
  - Body: `LoginBody` (`EmailStr`, `password` 1–256 chars).
  - Flow: `authenticate(db, email=str(payload.email), password=payload.password)`:
    1. `get_by_email` does case-insensitive match (`lower(email)`).
    2. `verify_password` against the stored passlib hash.
    3. Returns `None` on either failure, the route returns 401 `"Invalid credentials"`.
  - On success: `create_access_token(subject=str(user.id), role=..., client_id=...)`.
    The JWT has `sub`, `role`, `client_id`, `exp` claims.
  - Response: `{"access_token": "<jwt>", "token_type": "bearer"}`.

- **POST `/auth/logout`**
  - Auth: `Depends(get_current_user)`.
  - Flow: stateless — just asks the client to drop its token.
  - Response: `{"detail": "Logged out"}`.

### 4.7 Templates (`/templates`) — `app/api/routes/templates.py`

- `POST /templates` (Create) — `require_template_management` (superadmin OR enterprise
  admin with `Plan.can_manage_templates = TRUE`). Extra rules:
  - If `payload.is_global=true`: caller must be superadmin AND `client_id` must be None.
  - Otherwise `client_id` is required and, if caller is `ENTERPRISE_ADMIN`, must equal
    the caller’s `client_id`.
- `GET /templates` (Read all) — any authenticated; result is filtered by
  `list_templates_for_user`:
  - `BUSINESS_ADMIN` and `user` with `BUSINESS_ADMIN`-like plan see only global.
  - `ENTERPRISE_ADMIN`/`USER` with a `client_id` see global + their client.
  - `SUPER_ADMIN` (no `client_id`) sees everything.
- `GET /templates/{id}` — same rules but for a single template.
- `PUT /templates/{id}` — `require_template_management`. Flipping `is_global=true` is
  superadmin-only; when flipping, `client_id` is cleared to None.
- `DELETE /templates/{id}` — `require_template_management` (enterprise cannot delete
  global or other tenants’).
- `GET /templates/{id}/fields` — list fields. Enforces same role-aware visibility.
- `DELETE /templates/{id}/fields` — delete all fields in the template. Enterprise only
  for own non-global templates.

### 4.8 Fields (`/fields`) — `app/api/routes/fields.py`

- `POST /fields` — `require_template_management`. Body `FieldCreate`. Checks that the

  foreign.
- `GET /fields/template/{template_id}` — list fields with the same role rules as
  templates.
- `GET /fields/{field_id}` — read single, role-gated.
- `PUT /fields/{field_id}` — same rules as create.
- `DELETE /fields/{field_id}` — same rules.

### 4.9 Documents (`/documents`) — `app/api/routes/documents.py`

- **POST `/documents/upload`** (Create)
  - Auth: `get_current_user`.
  - Form fields: `template_id: int` (required), `client_id: int | None` (required for
    superadmin), `file: UploadFile`.
  - Flow:
    1. Resolve `final_client_id`: superadmin needs `client_id` in the form; everyone
       else must have `user.client_id`.
    2. `ensure_user_can_use_template(db, user, template_id)` — 403 if the template is
       not usable for this user's plan/tenant.
    3. `os.makedirs(settings.UPLOAD_DIR, exist_ok=True)`.
    4. Sanitize the filename: keep only the extension, generate `uuid4().hex + ext` so
       that two users cannot collide and the original filename (which could be
       path-traversal) is discarded.
    5. Write the file bytes to `UPLOAD_DIR/<uuid>.<ext>`.
    6. `create_document(db, client_id=final_client_id, template_id=..., file_url=save_path)`
       inserts a `documents` row with `status="uploaded"`.
  - Response: `DocumentOut`.

- **GET `/documents`** — list documents visible to the caller (tenant filter).
- **GET `/documents/{id}`** — read one; 403 if wrong tenant.
- **POST `/documents/{id}/process`** (Update + side effects)
  - Auth: `get_current_user` + tenant check.
  - Flow:
    1. Require `doc.status not in ("processing", "done")`.
    2. Require `doc.template_id is not None`.
    3. `set_document_status(db, doc, "processing")`.
    4. `run_document_processing(db, doc.id)` — see section 6 for the deep trace. Raises
       `ValueError` → 400, anything else → 502.
    5. On success: return `{"document": <DocumentOut>, "task_id": None,
       "status": "done", "extractions_created": <int>}`.
- **PUT `/documents/{id}/reprocess`** — same as process but without the `"done"` guard,
  so callers can re-run extraction.
- **GET `/documents/{id}/extractions`** — list `ExtractionOut` with `field_name` and
  `field_label` attached in memory from `ex.field`.
- **GET `/documents/{id}/extractions/summary`** — flat `{field_name: value}` JSON.
- **DELETE `/documents/{id}`** — deletes extractions, removes the file from disk if it
  exists (`os.remove(doc.file_url)`), then deletes the row.

## 5. Security Measures

### 5.1 Password hashing
- File: `app/core/security.py`.
- Code: `pwd_context = CryptContext(schemes=["pbkdf2_sha256", "bcrypt"], deprecated="auto")`.
- `hash_password(password)` runs Passlib with `pbkdf2_sha256` (default scheme).
- `verify_password(password, hashed)` verifies against either scheme (legacy bcrypt hashes
  still validate). `deprecated="auto"` flags legacy hashes for optional rehash, but the
  codebase currently doesn’t auto-upgrade on login — a known trade-off (section 10).
- Why `pbkdf2_sha256` default: pure-Python, no native dependency issues on Windows,
  tunable iteration count, widely reviewed. Bcrypt retained for legacy verification.

### 5.2 JWT authentication
- File: `app/core/jwt.py`.
- `create_access_token(subject, role?, client_id?)` sets `exp = now(UTC) + minutes` with
  `settings.ACCESS_TOKEN_EXPIRE_MINUTES` (default 60), encodes with `settings.SECRET_KEY`
  using `settings.ALGORITHM` (default `HS256`).
- `decode_token(token)` uses `jose.jwt.decode`, which verifies signature + expiration.
- Transport: `OAuth2PasswordBearer(tokenUrl="/auth/login")` (`app/api/deps.py`). Every
  protected route has `Depends(get_current_user)` so an invalid token returns `401
  Invalid or expired token`.

### 5.3 CORS
- File: `app/main.py`.
- Explicit allowlist: `http://localhost:5173`, `http://127.0.0.1:5173`,
  `http://localhost:5174`, `http://127.0.0.1:8000`.
- Dev regex additionally allows `localhost`, `127.0.0.1`, and private LAN
  `192.168.x.x` (any port) — needed so the dev server can be reached from a phone on
  the same network for mobile demos.
- `allow_credentials=True`, `allow_methods=["*"]`, `allow_headers=["*"]`.

### 5.4 Input validation / type safety
- Every request body is a Pydantic model (`app/schemas/`). For example:
  - `UserCreate.password: Field(min_length=8, max_length=72)` — length caps prevent
    DoS on passlib (bcrypt’s 72-byte input limit is respected). Emails use `EmailStr`
    so RFC-invalid emails are rejected early.
  - `LoginBody`: `ConfigDict(extra="ignore")`, password max 256 so clients can’t
    submit pathological inputs.
  - `FieldCreate` / `TemplateCreate` coerce all parameters.
- FastAPI returns 422 + field error list on violation.

### 5.5 SQL injection prevention
- All database access goes through SQLAlchemy ORM, which bind-parameterizes.
- The few places that use raw SQL (Alembic migrations and the two script files) all use
  `sa.text("... :name")` with dict params (`{"name": LB_TEMPLATE_NAME}`). No string
  interpolation into SQL.
- `backend/scripts/ensure_db.py` uses `cur.execute(sql, (param,))` with positional
  parameters; the only interpolated value is the database name for `CREATE DATABASE`,
  which is unavoidable (DDL can’t be parameterized) and is fed from the trusted
  `settings.DATABASE_URL`.

### 5.6 File upload hardening
- File: `app/api/routes/documents.py::upload_document`.
- Original filename is discarded. Only the extension is preserved:
  ```python
  ext = os.path.splitext(file.filename)[1].lower()
  safe_name = f"{uuid.uuid4().hex}{ext}"
  save_path = os.path.join(settings.UPLOAD_DIR, safe_name)
  ```
- This defeats directory traversal (`../../etc/passwd`) and name collision.
- Files are saved under `settings.UPLOAD_DIR` (default `uploads/`, gitignored).

### 5.7 OCR path hardening
- `app/services/ocr.py::_resolve_file_path(file_url)` refuses empty paths, and for
  absolute paths it also tries a fallback to `backend/uploads/<basename>` so that moving
  the project doesn't leak absolute host paths.

### 5.8 Tenant isolation (authorization)
- Every read/write route that touches tenant data does an equality check
  `doc.client_id == user.client_id` (or equivalent). `SUPER_ADMIN` is the only role
  that bypasses it. See lines 81–83, 100–101, 140–142, 166–168, 191–193, 212–214,
  230–232, 246–248 of `app/api/routes/documents.py` — identical pattern across
  `extractions.py`, `payments.py`, `subscriptions.py`.
- Template visibility is centralized in `ensure_user_can_use_template` and
  `list_templates_for_user`.

### 5.9 Role enforcement
- `require_superadmin` and `require_template_management` are FastAPI dependencies so
  they run *before* the route body executes (FastAPI’s dependency resolution).
- Role comes from the JWT (not query params), so spoofing requires breaking JWT.

### 5.10 XSS / CSRF
- XSS: no server-side HTML rendering; responses are JSON. React auto-escapes interpolated
  strings, and the frontend never uses `dangerouslySetInnerHTML` on user input.
- CSRF: stateless JWT in `Authorization: Bearer <token>` header (not cookies). Browsers
  don’t auto-attach bearer headers cross-origin, so CSRF is a non-issue for this auth
  model.

### 5.11 Request logging / auditing
- `app/middlewares/api_logging.py::APILoggingMiddleware` reads the body (max 10 000
  bytes), attempts JSON decode, and inserts an `api_logs` row with method+path,
  `status_code`, `client_id` (from JWT if present), `request_payload`. It never blocks
  the response on logging errors (`except Exception: db.rollback() ... finally db.close()`).

### 5.12 Secret management
- Secrets are pulled from `.env` via pydantic-settings. None are hard-coded except the
  placeholder `SECRET_KEY="CHANGE_ME"` default — which is a known trade-off for ease of
  local dev; production is expected to override it in the environment.

### 5.13 Environment variables (security-relevant)

### 5.14 Authentication & authorization flow — login to protected route

1. User submits `{email, password}` to `POST /auth/login` (`routes/auth.py:110`).
2. `authenticate(db, email, password)` (`crud_user.py:63`):
   - `get_by_email` uses `or_(func.lower(User.email) == normalized,
     func.lower(func.trim(User.email)) == normalized)` → returns the row if found.
   - `verify_password(password, user.hashed_password)` uses passlib (`security.py:12`).
3. On success, the route calls `create_access_token` (`core/jwt.py:6`) and returns
   `{"access_token": ..., "token_type": "bearer"}`.
4. Frontend stores the token in `localStorage` under `autodoc_access_token`
   (`AuthContext.jsx:3`).
5. For each protected request, the frontend (not shown everywhere in code yet; the
   marketing pages don’t call protected endpoints) is expected to send
   `Authorization: Bearer <token>`.
6. FastAPI routes declared with `Depends(get_current_user)` enter
   `app/api/deps.py:14`:
   - `oauth2_scheme(...)` extracts the bearer token from the `Authorization` header.
   - `decode_token` verifies signature + exp; on any failure → 401.
   - Loads `User` by `int(payload["sub"])`.
7. If the route uses `require_superadmin` or `require_template_management`, the extra
   checks run.
8. Route body executes; returns JSON. Middleware logs the request.

## 6. End-to-End Pipeline (Step-by-Step)

The trace below follows the single most important user action in the app:

> "A business admin uploads a PDF, runs extraction, and sees the results."

### Step 1 — UI click handler (frontend)

- File: `frontend/src/pages/public/Signup.jsx` (if signing up) or the user’s
  business/enterprise dashboard (placeholder today). In the actual E2E automated test
  (`backend/scripts/e2e_upload_lebanese_pdf.py`), the flow is equivalent:
  1. POST `/auth/signup/business` with `{organization_name, company_name, full_name,
     email, password, client_type: "business"}`.
  2. POST `/auth/login` with `{email, password}` → stores `access_token`.
  3. GET `/templates` to list visible templates and pick one (e.g. the Arabic LB ID).
  4. POST `/documents/upload` multipart with `template_id` and `file`.

### Step 2 — Login handler (frontend)

- File: `frontend/src/pages/public/Login.jsx`, `handleSubmit` (lines 33–89).
  - Fetch to `${API_BASE_URL}/auth/login` with `{email, password}`.
  - On HTTP 200: `loginWithToken(token)` → `AuthContext.jsx:71`.
  - `loginWithToken` decodes the JWT payload locally (`decodeToken` + `getSessionFromToken`),
    stores the token under `autodoc_access_token`, sets the context `session`.
  - `navigate(getRoleHome(session.role), { replace: true })` — `ROLE_HOME` maps roles
    to dashboards.

### Step 3 — Backend login route

- File: `backend/app/api/routes/auth.py:110`:
  - `payload: LoginBody` runs Pydantic validation (EmailStr + password length).
  - `authenticate(...)` (`crud_user.py:63`) → passlib verify → returns `User`.
  - `create_access_token(sub=str(user.id), role=..., client_id=...)`.

### Step 4 — Upload request (frontend → backend)

- Frontend (once built) will POST multipart/form-data to `/documents/upload` with the
  JWT in `Authorization: Bearer <token>`.
- Backend middleware chain runs in order:
  1. `APILoggingMiddleware.dispatch` (`middlewares/api_logging.py:18`): reads body (if
     JSON), decodes JWT for `client_id`, calls the route, then inserts an `api_logs` row
     with the final status code.
  2. `CORSMiddleware`: validates Origin / adds headers.
  3. FastAPI routing resolves `POST /documents/upload` to
     `routes/documents.py:30::upload_document`.

### Step 5 — Upload handler (backend)

- `routes/documents.py:30`:
  1. FastAPI evaluates deps: `get_db()` yields a Session, `get_current_user(...)` parses
     the JWT and loads the User.
  2. `user.role` checked; `client_id` derived.
  3. `ensure_user_can_use_template(db, user, template_id)` — 403 if template is not
     permitted (e.g. business trying to use a tenant-only template).
  4. UUID filename generated; file written to `uploads/<uuid>.<ext>`.
  5. `create_document(db, client_id, template_id, file_url)` inserts a `documents` row
     with `status="uploaded"`.
  6. Returns `DocumentOut`.

### Step 6 — Process request

- Frontend posts to `/documents/{id}/process`.
- `routes/documents.py:88::process_document`:
  1. `get_document(db, id)` → load or 404.
  2. Tenant check.
  3. Refuses if already `processing` or `done`.
  4. `set_document_status(db, doc, "processing")` updates the row and commits.
  5. Calls `run_document_processing(db, doc.id)` (`services/document_pipeline.py:32`).
  6. If `ValueError`: 400; any other exception: 502 (and `set_document_status(..., "failed")`
     was already called inside the pipeline’s except block).
  7. Reloads the doc and returns `DocumentProcessOut`.

### Step 7 — The pipeline internals (`services/document_pipeline.py:32::run_document_processing`)

1. `get_document(db, document_id)` — 2nd load (session boundary).
2. `delete_extractions_for_document(db, doc.id)` — wipe any stale rows (useful for
   reprocessing).
3. `get_extraction_context(db, document_id)` (`services/extraction_context.py:20`):
   - Loads `Document`, `Template`, and all `Field` rows.
   - Returns `{template_id, template_name, template_description, fields: [...]}` where
     each field has `{id, name, label, field_type, description, required}`.
4. `get_text_and_structured_ocr_from_pdf(doc.file_url)` (`services/ocr.py:235`):
   a. `_resolve_file_path(file_url)` — supports absolute, relative, or
      `app/uploads/...` paths.
   b. `_open_document_for_ocr(file_path)`:
      - If extension is `.pdf`, open as PDF with PyMuPDF.
      - Otherwise, try PyMuPDF (works for many PNG/JPEG). If 0 pages or error, convert
        via PIL (`_image_file_to_temp_pdf`): open with Pillow → `_pil_to_rgb` (flatten
        transparency) → `_downscale_pil_for_ocr` (cap largest edge at
        `AUTODOC_OCR_MAX_IMAGE_EDGE`, default 2400 px) → save as PNG, then insert into
        a fresh `fitz.Document` and save to a temp `.pdf`.
   c. For each page:
      - `page.get_text()` — if the embedded text length ≥
        `PDF_EMBEDDED_MIN_CHARS_TO_SKIP_OCR` (40), skip Vision and use the embedded
        text directly (fast path for text PDFs).
      - Otherwise, `page.get_pixmap(dpi=PDF_OCR_DPI=300)` → PNG bytes → PIL → downscale
        → `_ocr_image_google` → Google Cloud Vision `document_text_detection`.
      - Vision response is parsed: `_google_annotation_to_lines` walks
        `pages[].blocks[].paragraphs[].words[].symbols[]` to build line-level objects
        `{text, confidence}` where confidence is the average of word confidences.
   d. Returns `{"text": full_text, "pages": [{page_index, source, lines: [...]}, ...]}`.
5. `clean_ocr_text(ocr_text)` (`services/text_cleanup.py:172`):
   - `_normalize_line_breaks` — unify \r\n, \r, \n, collapse 3+ newlines to 2.
   - `_normalize_whitespace` — single space for multiple spaces/tabs.
   - `_remove_non_printable` — keep printable + `\n` + `\t`, drop Unicode category "C".
   - `_fix_common_ocr_character_errors` — merges broken digits (including Eastern-Arabic
     digits U+0660..U+0669), fixes stray `|~\``, tidies punctuation spacing, joins
     broken currency.
   - `_fix_grammar_and_spacing_errors` — adds missing space after punctuation, uppercase
     after sentence end, fixes hyphenated words.
   - `_remove_ocr_artifacts` — drops 3+ dashes/underscores, isolated symbols, excessive dots.
   - `_strip_whitespace` — trim each line and the whole blob.
6. `extract_with_llm(cleaned_text, context)` (`services/llm_extraction.py:70`):
   - Raises if `OPENAI_API_KEY` missing.
   - `_build_messages` composes:
     - system: "You are an expert at extracting structured data from documents..."
     - user: lists every field as `- name (type) (required): description`, then includes
       the full cleaned text between `---` markers, asking for a raw JSON object.
   - `client.chat.completions.create(model=settings.OPENAI_MODEL, messages=messages)`.
   - Catches `openai.RateLimitError`, `AuthenticationError`, `APIError` and converts
     them into `ValueError` (so the route maps them to 400/502).
   - `_parse_json_from_response` strips ```json ``` fences if present and runs
     `json.loads`. Returns the dict.
7. `calculate_confidence(extraction_result, context)` (`llm_extraction.py:331`):
   - For each field in context: if value is empty → 0.0 (required) or 0.1 (optional).
     Else detect ambiguity (`?`, "maybe", "N/A", very short, `A or B`) → 0.4.
     Else `_validate_field_by_type(value, field_type)`:
     - `date`: regex matches, year sanity check, 0.6–0.9.
     - `number`/`currency`/etc.: cleans symbols, accepts int/decimal/commas, 0.9.
     - `email`: basic regex, 0.9; `@` but invalid → 0.4.
     - `phone`: 10+ digits allowing formatting, 0.85–0.9.
     - `text`/unknown: 0.85 if non-empty, else 0.
8. `detect_low_confidence_fields(confidence_dict, threshold=0.5)` — logged as a warning.
9. Loop over each field in the template:
   - `value = extraction_result.get(field_name)`, `confidence =
     confidence_dict.get(field_name, 0.5)`.
   - `create_extraction(db, document_id=doc.id, field_id=field.id, value_text=...,
     value_json=value, confidence=confidence)` — persists one Extraction row per field.
10. `set_document_status(db, doc, "done")` — sets `doc.processed_at = datetime.utcnow()`.
11. On any exception inside the try, `set_document_status(db, doc, "failed")` is called
    and the exception is re-raised.
12. Returns the number of extractions created (int).

### Step 8 — Response to the frontend

- The route returns `DocumentProcessOut` (includes the full updated `DocumentOut`,
  `task_id=None`, `status="done"`, `extractions_created=<n>`).
- Frontend would then call `/documents/{id}/extractions/summary` for a flat dict or
  `/documents/{id}/extractions` for the list + metadata.

### Step 9 — Middleware post-processing

- Back in `APILoggingMiddleware.dispatch`, after `call_next` returns, the row is
  inserted into `api_logs` with `status_code=200`, `client_id` from the JWT, and the
  initial body payload.

### Step 10 — Frontend state update

- For the current marketing site: the login page flips `session` in the `AuthProvider`
  and `navigate()`s. The dashboards are placeholders, so there is no live list refresh
  yet; the product requires implementing API calls in `B_api.jsx` etc. (see section 10).

## 7. Frontend Logic Breakdown

### 7.1 State management
- **Global auth**: React Context in `src/context/AuthContext.jsx`. Exposes
  `{session, role, token, isAuthenticated, loading, loginWithToken, logout,
  restoreSession}`. Context is chosen (not Redux/Zustand) because there is only one
  piece of cross-cutting state — the session — and React Context is enough.
- **Local page state**: plain `useState`. For example, `Login.jsx` manages
  `{formData, showPassword, loadingForm, error}`. `Signup.jsx` also manages a
  `currentPlan` toggle synced to the URL query param.
- **UI global state**: `SuperNav.jsx` keeps local state for `openMega`, `isMobileOpen`,
  `theme` (persisted to `localStorage` as `"theme"`), `isProfileOpen`, and scroll-based
  visibility flags.

### 7.2 Routing
Defined in `src/App.jsx` via `react-router-dom` v7:

Public routes:
- `/` → `pages/public/homepage`
- `/about` → `pages/public/about`
- `/pricing` → `pages/public/Pricing`
- `/contact` → `pages/public/Contact`
- `/signup` → `pages/public/Signup`
- `/login` → `pages/public/Login`

Protected routes (via `<ProtectedRoute allowedRoles=[...]>`):
- `super_admin` → `/super`, `/super/clients-plans`, `/super/templates-ai`,
  `/super/monitoring`.
- `enterprise_admin` → `/enterprise`, `/enterprise/template`, `/enterprise/api`,
  `/enterprise/profile`.
- `business_admin` + `user` → `/business`, `/business/api`, `/business/profile`.

Route guard (`components/ProtectedRoute.jsx`):
```jsx
if (loading) return null;
if (!isAuthenticated) return <Navigate to="/login" replace state={{from}}/>;
if (allowedRoles?.length && !allowedRoles.includes(role))
  return <Navigate to={getRoleHome(role)} replace />;
return <Outlet />;
```

### 7.3 API integration
- No dedicated client module yet. Pages use `fetch` directly with
  `API_BASE_URL = import.meta.env.VITE_API_BASE_URL || 'http://127.0.0.1:8000'`.
- Example (`Login.jsx:46`):
  ```js
  const response = await fetch(`${API_BASE_URL}/auth/login`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ email: formData.email.trim(), password: formData.password }),
  });
  ```
- Error handling: defensive `await response.json().catch(() => ({}))`. FastAPI validation
  errors come as an array in `data.detail`; the code joins them into a readable string
  so the user sees "email: field required" instead of `[object Object]`.
- Networking failures fall through to `catch (err)` and show
  "Unable to reach the server. Please try again.".

### 7.4 Component hierarchy (public/landing)

```
Homepage
 └─ Bg
    ├─ Navbar
    ├─ Hero
    ├─ Features
    ├─ HowItWorks
    ├─ UseCases
    ├─ CTA
    └─ Footer

Login / Signup / About / Pricing / Contact
 ├─ Navbar
 ├─ ...page content...
 └─ Footer
```

No props drilling beyond a few props (`SignupForm` receives `plan`, `onSwitchPlan`).
Cross-cutting auth state goes through `useAuth()` context instead.

### 7.5 Custom hooks
- `useAuth` (`context/AuthContext.jsx:101`) — wraps `useContext(AuthContext)` and
  throws if used outside the provider. This fail-fast pattern catches misuse early.
- Utility `getRoleHome(role)` isn’t a hook but is co-located.

### 7.6 Form handling
- No form library; `useState` + native HTML `required`/`minLength`.
- Client-side validations implemented in `handleSubmit` (e.g. `Signup.jsx:73–82`:
  password length + match check) before hitting the backend.
- Errors render in a dedicated `.form-error` or `.error-message` div; success states
  swap the whole form for a confirmation card (see `Signup.jsx:118` and
  `Contact.jsx:76`).

### 7.7 JWT handling on the client
- `AuthContext.decodeBase64Url` adds padding before `atob` to handle URL-safe base64.
- `getSessionFromToken` rejects:
  - Tokens that don't split into 3 parts.
  - Tokens whose payload doesn’t JSON-parse.
  - Tokens without `role` or whose `role` isn’t in `ROLE_HOME`.
  - Expired tokens (`now >= payload.exp`).
- This means even if `localStorage` contains a stale token, the user is bounced to
  `/login` on page reload.

only test-like code is in `backend/scripts/`:

- `e2e_upload_file.py` — opens an existing PDF on disk, runs the full flow (signup →
  login → upload → process → summary) against a live running server at
  `http://127.0.0.1:8000`. Useful for smoke testing OCR/LLM on arbitrary files.
- `e2e_upload_lebanese_pdf.py` — builds a small Arabic-ID PDF in memory using PIL and
  PyMuPDF and runs the same flow. Reports the per-field extractions to stdout.
- `run_pipeline_once.py` — runs `run_document_processing` directly against the DB,
  bypassing the HTTP layer. Writes a JSON summary (non-ASCII safe) to
  `scripts/pipeline_last_result.json` — useful on Windows where the console can’t
  print Arabic reliably.
- `fix_fields_columns.py` — a self-healing schema patcher that adds `fields.label` and
  `fields.description` if they’re missing.
- `ensure_db.py` — creates the database if missing.

These scripts depend on the live API and a seeded template (Lebanese ID or CV), so they
are effectively manual integration tests.

What would be needed to complete test coverage:
- Pytest suite for `app/crud/*` (pure DB logic).
- FastAPI `TestClient` suite for auth, tenant isolation, permissions, upload/process.
- Mock clients for OpenAI and Google Vision.
- A frontend test runner (Vitest) for `AuthContext` token decoding edge cases and
  `ProtectedRoute` role-matching behavior.

## 10. Known Trade-offs & Future Improvements

### Trade-offs (as-built)

1. **Pipeline is synchronous**. `/documents/{id}/process` calls
   `run_document_processing` inline; OCR + LLM take several seconds and block the
   worker. `settings.CELERY_BROKER_URL/CELERY_RESULT_BACKEND` exist but Celery is
   **not wired**. For the MVP this was accepted to keep the code simple; in production
   this should move behind a task queue so the route returns immediately with a real
   `task_id`.
2. **Refund is a hard delete**. `/payments/{id}/refund` (`routes/payments.py:63`) calls
   `delete_payment`. A real refund should create a negative/refund row and keep the
   audit trail.
3. **Password rehash on login is not automatic**. `passlib` is configured with
   `deprecated="auto"`, but the login code doesn’t call `pwd_context.needs_update` +
   rehash. Easy to add in `authenticate()`.
4. **JWT blacklisting / logout is stateless**. `/auth/logout` just asks the client to
   drop the token; a stolen token remains valid until `exp`. A Redis blacklist or a
   short `exp` + refresh-token pair would mitigate this.
5. **CORS is permissive in dev**. The regex `192.168.x.x:*` is for mobile demos; it
   should be disabled for production.
6. **`SECRET_KEY` default is `"CHANGE_ME"`**. Helpful for local setup but must be
   overridden via `.env` before anything reachable from the internet is deployed.
7. **No rate limiting**. Anyone can hammer `/auth/login` or `/documents/upload`.
   Adding `slowapi` or an Nginx layer in front is needed pre-production.
8. **Tenant quotas are partial**. `Plan.max_users` is enforced; there’s no limit on
   documents processed per month, per second, etc.
9. **Auto-commits inside pipeline**. `create_extraction` commits per field — a long
   pipeline writes N rows with N commits. Batching into a single commit would be
   faster but requires re-plumbing the CRUD layer.
10. **`placeholder` protected pages**. `Sdashboard`, `B_dashboard`, `EDashboard`, etc.
    render only static text. They are routed and protected, but the actual features
    (upload UI, extractions viewer, template builder, plan admin, monitoring) need to
    be built on top of the existing API.
11. **Legacy columns in models were commented out** rather than removed: see
    `models/user.py`, `models/client.py`, `models/plan.py`, `models/document.py`.
    The legacy ownership link (`owner_user_id`) was dropped in migration
    `d4c49f63235c` and the ORM field was commented out but left in the file for
    historical context.
12. **API logs never capture response payloads** — the middleware sets
    `response_payload=None` to keep latency/size under control. A future improvement
    is to sample large responses with a size cap.
13. **No dedicated error type layer**. The pipeline uses generic `ValueError` and
    `Exception`; a typed hierarchy (e.g. `OcrError`, `LlmError`, `TemplateError`)
    would allow the route to map to clean 4xx/5xx codes without string matching.
14. **Two-step upload+process UX**. The frontend must call `/documents/upload` and then
    `/documents/{id}/process`. A `?autostart=true` option or a single `/process` that
    accepts a file would cut latency and complexity.
15. **File storage is local disk**. `UPLOAD_DIR` defaults to `uploads/`. For production,
    this should be S3 (or any compatible blob store). The `file_url` column is already
    large enough (2048 chars) to hold presigned URLs.
16. **Country/industry migration (`20260306_country_industry.py`) is a no-op
    placeholder** in source because the schema change was applied directly to the DB
    before being committed. That’s a documented gap; the actual columns were not
    recovered into code.

### If I had more time
- Background tasks via Celery + Redis (already in config but unused), so
  `/documents/{id}/process` returns a `task_id` and the frontend polls.
- Full React UI for: template builder, field editor, document list with per-row status,
  extractions viewer with confidence highlighting (< 0.5 surfaced), super admin
  client/plan management, monitoring dashboard pulled from `api_logs`.
- Stripe wired to `/subscriptions` via Stripe Checkout Sessions + webhooks instead of
  the current mock payment row.
- Pytest + Vitest test suites with coverage target (> 80% on `crud/*` and
  `services/*`).
- SOC-2-friendly audit: request/response payload scrubbing, PII redaction, retention
  policies on `api_logs`.
- Refresh tokens + short-lived access tokens.
- Rate limiting (slowapi) on `/auth/login` and `/documents/upload`.
- Dockerfile + docker-compose for Postgres + backend + frontend, plus a GitHub Actions
  workflow that runs lint, tests, and `alembic upgrade --sql` as a smoke check on PRs.
- Migration for `country`/`industry` on `clients` so the schema matches what the
  placeholder migration implies.
- Structured error types (`OcrError`, `LlmError`, `TemplateError`, `QuotaError`) to
  avoid generic `except Exception`.
- Field-level unique constraint on `subscriptions` (partial unique index on
  `(client_id) WHERE status='active'`) so the DB enforces "one active subscription per
  tenant" instead of the route.
- Migrate `plans.monthly_price` to `NUMERIC(12,2)` or keep `INT cents` but enforce it
  at the schema level with a check constraint.
- Replace file disk storage with S3/GCS + presigned URLs.
